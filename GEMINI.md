# Gemini CLI System Rules for WHOOP Stats

This document provides contextual instructions and project conventions for Gemini CLI. **These are foundational mandates and take absolute precedence.**

## Project Overview
`whoop-stats` is a high-performance, dual-mode (webhook & poller) ingestion engine and dashboard for WHOOP fitness data. It is heavily optimized for both cloud environments and 24/7 homelab deployments with consumer SSDs.

## Technology Stack & Conventions

### Backend (Go)
*   **Version:** Go 1.22+
*   **Routing:** `go-chi/chi` for REST endpoints.
*   **Database Access:** `sqlc` for fully type-safe database queries.
    *   **Rule:** ALWAYS run `sqlc generate` in the `whoop-stats` directory after modifying any `queries/*.sql` or `migrations/*.sql` files.
    *   **Rule:** DO NOT manually edit files in `internal/db/`. They are generated by `sqlc`.
*   **Logging:** `log/slog` for structured, JSON-based logging.
    *   **Rule:** Respect the `LOG_LEVEL` environment variable. Do not add excessive `Info` or `Debug` logs inside tight loops to prevent Write Amplification (WA) on user SSDs.
*   **Concurrency:** Use goroutines for background tasks (like polling and webhook processing) but protect state with proper mutexes or Postgres advisory locks where necessary.

### Frontend (Next.js)
*   **Framework:** Next.js 16+ using the **App Router** (`app/` directory).
*   **Styling:** Tailwind CSS v4.
*   **Components:** React Server Components (RSC) are preferred for initial data fetches to minimize layout shift. Client components (`"use client"`) should be used strictly for interactivity (e.g., charts, maps).
*   **API Client:** We use `openapi-fetch` and `openapi-typescript`.
    *   **Rule:** The frontend data structures are directly tied to the Go backend via the Swagger/OpenAPI spec (`docs/openapi.json`). If Go structs change, ensure `docs/openapi.json` is updated so the frontend remains type-safe.

### Database (PostgreSQL / TimescaleDB)
*   **Core:** PostgreSQL 15 + TimescaleDB extension.
*   **Design:** Time-series data (cycles, sleeps, workouts, recoveries) MUST be stored in TimescaleDB `hypertables` partitioned by `start_time`.
*   **Aggregations:** Use TimescaleDB Materialized Views (`Continuous Aggregates`) for high-level dashboard queries (e.g., 30-day strain/recovery trends) instead of computing them on the fly.
*   **Schema Changes:** All schema changes must be done via standard `.sql` migration files in the `migrations/` folder.

## Core Architectural Mandates

1.  **Dual Ingestion Engines:** The app must always support both Webhook mode (for immediate pushes) and Poller mode (for homelabs behind NAT). 
2.  **Webhook Inbox Pattern:** Webhook endpoints must respond with `200 OK` instantly. Payload processing must be asynchronous (stored in `webhook_events` and processed by a background worker) to ensure data integrity during API slowdowns.
3.  **Idempotency:** All database `UPSERT` operations must handle conflicts gracefully (`ON CONFLICT (id, start_time) DO UPDATE ...`). No duplicate data should ever be recorded.
4.  **SSD Protection:** Assume this runs on consumer SSDs. Minimize write amplification by maintaining `tmpfs` mounts for `/tmp` and `/var/log` in Docker, and using the `local` logging driver.

## Security
*   **Credentials:** NEVER log, print, or commit `WHOOP_CLIENT_SECRET`, `ENCRYPTION_KEY`, or any OAuth tokens (`.whoop_token.json`).
*   **Encryption:** OAuth tokens must always be encrypted using AES-256-GCM before resting in the database.
